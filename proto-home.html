Compose your own workflow

Four commands, clone to a workflow built out of your own words. The engine is pure standard library — no third-party deps to install.

1 · Clone & install

git clone https://github.com/mtichikawa/workflow-engine.git
cd workflow-engine && pip install -e .

2 · Give it a brain (pick one)

# Claude CLI — free on a Max plan, and the default. Install it, sign in, done.
# — or the billed API (~cents/run):
pip install anthropic && export ANTHROPIC_API_KEY=sk-ant-… && export ENGINE_BRAIN=api

3 · Compose one of ours

python -m engine.cli compose "read GitHub issues, label and prioritize them, and draft a first response"

It prints the recipe it built — each step tagged, plus whether it's runnable:

Composed recipe for: 'read GitHub issues, label and prioritize them, and draft a first response' reuse classify reuse rank reuse respond reuse verify reuse act [GATE] runnable: True

Now your turn

Describe any judgment-heavy, multi-step job:

python -m engine.cli compose "<your workflow in plain English>"

4 · Run what it composed

python -m engine.cli compose "screen support tickets and draft replies" \
    --run --input "API returns 500 on every POST since this morning" --auto

Stages a result to state/composed-<run_id>.json. Hit a gap? Add --draft to fill it with a provisional specialist (untrusted — check it at the gate before you rely on it).

Run a workflow yourself

Same clone + brain as the Composer steps. Then run any of the three example recipes — and point them at your own input.

Run the triager on a real repo

python -m engine.cli run triage --repo vercel/next.js --limit 5
inputreal open issues pulled straight from the repo doesclassify · rank urgency · route to a component · draft a reply · verify · stage you geta board summary in your terminal, plus a printed run id wherefull log → state/<run_id>.json  ·  staged actions → output/staged.jsonl

The other two recipes

python -m engine.cli run content --topic "why flaky tests erode trust" \
    --brief "sharp technical founder" --auto
python -m engine.cli run refine  --topic "why on-call rotations burn people out" --auto

refine loops draft → grade → rewrite until it clears the bar; content researches, writes, and checks it's on-brand.

Now your turn

Point it at any public repo, or any topic:

python -m engine.cli run triage --repo psf/requests --limit 8

Nothing is ever sent — act is gated and only stages. Re-inspect any run with python -m engine.cli board <run_id>.

The third pillar · alongside the Composer & the Dispatcher

A shared library of specialists

Every recipe is built from the same pool of single-purpose specialists — each does exactly one job, with one input→output contract. The Composer doesn't write new code per task; it reuses what's already here. So the library is the part that compounds: the same specialist does real work across completely different jobs.

Reuse across the four example recipes

verify and act run in all four — the same judge, the same gate, pointed at a triaged issue, a drafted post, a self-graded rewrite. Build it once, reuse it everywhere.

The pool — nine specialists, each one job

classify
Sort an item into one of a fixed set of categories.
item → label
rank
Score an item for urgency or quality.
item → score
verify
Judge whether an output meets a stated standard.
subject → verdict
route
Assign an item to the right component or owner.
item → component
respond
Draft a reply in a given tone.
item → reply
review
Critique a draft against criteria.
draft → critique
write
Produce a piece of writing to a brief.
brief → draft
fetch
Pull source material for a topic.
query → sources
act
Stage the final action behind a human gate.
payload → staged
shared capabilitydomain stephuman gate
When a job needs something the library lacks, the Composer doesn't fake it — it flags the gap and drafts a provisional stub (untrusted until it's trained on real examples). That's how the library grows without ever pretending to a capability it doesn't have.
