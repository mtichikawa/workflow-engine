See how it works by doing

Clone it and run a real workflow

No account, no key to start — the engine is pure Python and the default brain is the free Claude CLI. Three example recipes; each takes real input and stages a result for a human. Nothing is sent automatically.

 your terminal
git clone https://github.com/mtichikawa/workflow-engine
cd workflow-engine && pip install -e .

# a support inbox that reads itself — real GitHub issues in, staged actions out
python -m engine.cli run triage --repo vercel/next.js --limit 5

# research + write a short post in a set voice
python -m engine.cli run content --topic "why flaky tests erode trust" --auto

# draft → self-grade → rewrite until it passes (the loop)
python -m engine.cli run refine --topic "why on-call burns people out" --auto

Each run prints a board summary and saves to state/<run_id>.json; staged results land in output/. Point --repo at any public repo or swap the --topic — it's real data, not ours.

    Zoom 1 — compose & validate

    Composing parts into a workflow

    Every specialist declares a fixed contract — its exact inputs and outputs — so any part snaps into any recipe (a graph of steps, with branches and loops) without glue code. The composer wires the graph, and the same shared parts get reused across very different problems.

    modularity, shown

    Three real workflows reuse the same shelf, unchanged — and a fourth you define yourself reuses what fits and drafts the rest.

    A specialist glows when it's used by the recipe you hover. classify · rank · verify · act are shared; fetch · write · route · respond are recipe-specific. Hover your use case — it reuses what's on the shelf and marks the new parts (?) the composer would draft.

    the composer, benchmarked

    Given only a plain-English description, the composer re-wrote specialists I'd hand-built — scored against their real, human-labeled evals, it matched the originals. And when it can't cover a use case it flags the missing specialist and refuses to call the recipe runnable (2 of 2 coverable ran; 3 of 3 with gaps held back). It doesn't fake capability.

    the validator — checked before it runs

    A composed graph isn't trusted blindly. A static validator checks it first — entry/exit, reachability, loop termination (a backward edge must have a guard), dangling outputs, and contract compatibility. It's a little type-checker for the workflow: invalid graphs are rejected up front, not discovered halfway through a run.

    Zoom 2 — the dispatcher, live

    Controlled chaos

    The board is the shared surface the dispatcher writes onto. Work streams in and flows through the graph concurrently, each item advancing on its own; a specialist clones per item (the stacked chips); when verify flags a draft it loops back to rewrite (the red curve); the rest wait at the gate for a human, then flow out. Deliberately dumb, single-writer, concurrency-safe by construction — the piece I'm most proud of. The flow here is illustrative.

    items flow in continuously · specialists clone under load · all pause at the gate

    The atom, and how it learns

    how the learning works

    Retrieval-injected few-shot from curated examples — no fine-tuning. A drafted specialist is provisional until its eval clears a bar, then trusted. Examples are isolated per tenant by path — and building that seam surfaced a real bug: ThreadPoolExecutor doesn't propagate context, silently breaking tenancy under concurrency. Found and fixed. A finding fell out too: few-shot helps “mapper” specialists but hurts the checker.

    It generalizes — three distinct domains

    One engine, real data

    The same specialists compose into different recipes. Reuse between the first two is the compounding library; the third adds a control-flow loop.

    technical support

    Triage — a support inbox that reads itself

    A raw bug report comes in. The engine works out what it is, how urgent, and who owns it, drafts a first reply, checks that reply is any good — then stops and waits for a person to hit send.

    classify→rank→route→respond→verify→act ⏸
    creative

    Content — a writer that does its own research

    Give it a topic. It pulls real sources, picks an angle, writes a short post in a set voice, and checks it's on-brand before staging. Same classify / rank / verify / act as triage — only the writing step is new.

    fetch→classify→rank→write→verify→act ⏸
    quality control · with a loop

    Refine — an editor that won't ship slop

    It writes a draft, grades it against a strict bar (real specifics, no hype), and — if it falls short — rewrites and re-checks until it passes. On a real run it failed twice, revised, and passed on the third — the same loop-back mechanism the board animates — then waited for a human to publish.

    Honest frame · positioning

    The components aren't novel — orchestration, few-shot retrieval, human-in-the-loop labeling, and LLM-as-judge all exist (LangGraph, DSPy, Snorkel). The value is a coherent, from-scratch, working synthesis with measured results, and the hard parts solved by hand. Differentiated and forward-looking. Not novel — that distinction is the whole point.

    A complement to my shipped product (Microclaw), not an equal to it. For “can you ship production AI,” that product leads.
